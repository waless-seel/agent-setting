# review-thinking

現在のセッションで行った思考・判断・コマンド実行を振り返る
LLMの思考内、又は結果から決定的に再現できるものをコード化する
以下の2ファイルを生成する

| 成果物 | 説明 |
|--------|------|
| `thinking.md` | 思考プロセス・判断根拠・学習内容のドキュメント |
| `reproduce.{ext}` | セッションで用いた言語・環境に応じた、思考、作業を再現するスクリプト（Bash/Python/PS1等） |

## 出力先

```
{プロジェクトルート}/reviews/
└── {YYYY-MM-DD}-{セッション概要スラッグ}/
    ├── thinking.md
    └── reproduce.{ext}
```

`--dest` オプション指定またはプロジェクト内 `.claude/review-thinking.config` の設定がある場合は、
生成後に `{dest}/{slug}/` にも同内容をコピーする。

## ワークフロー

Todoリストを作成して、以下の各ステップを順番に実行すること。

### Step 1: 証跡収集とセッション振り返り

#### Step 1a: 証跡収集（Bash実行）

以下を実行して実際の変更履歴を一次情報として取得する:

```bash
git log --oneline -20 --format="%H %an %s"
```

取得したコミット一覧について、以下の基準でLLM実行コミットを判別する:

**判別基準（いずれかを満たせばLLM実行）:**
1. `git log` の `%an`（author name）が AI エージェント名（例: `Claude`、`GitHub Copilot` 等）
2. コミットメッセージに Co-Authored-By が含まれる

```bash
# author 名の確認（git log --format="%H %an %s" の出力で判断）
# Co-Authored-By の確認（必要に応じて）
git show {hash} --format="%B" --no-patch | grep -e "Co-Authored-By"
```

- **LLM実行コミット** → author 名が AI エージェント名、または Co-Authored-By あり
- **人間実行コミット** → 上記に当てはまらない（`要人間判断` として記録）

> **注意**: Co-Authored-By の付与はツールやルール設定に依存するため必須ではない。
> author 名を主判別基準とし、Co-Authored-By は補助的に参照する。

#### Step 1b: 会話コンテキストとの照合

証跡収集の結果を踏まえ、会話コンテキストを元に以下を整理する:

- **セッションの目標・背景**: ユーザーが最初に何を依頼したか
- **実行したアクションの一覧**: 証跡（git log）+ 会話コンテキストから統合
  - LLMのアクション: 会話コンテキストから取得
  - 人間のアクション: Co-Authored-By なしのコミット等、証跡から補完
- **各アクションの判断理由**: なぜそのアプローチを選んだか
- **各アクションの不確実性・推測箇所**: 確認せず推測で進めた点・判断に迷った点・情報不足だった点（意思決定ログの「自信度」「推測に頼った箇所」を埋める際の事前整理になる）
- **推測から実測への修正**: テスト実行・エラー観察・出力確認などで「推測が覆った」箇所を記録する（例: 期待値をテスト前後で変更した場合、その理由と変更前後を対比して残す）
- **発生したエラーと対処**: ハマった点・試行錯誤・解決策
- **最終的な成果**: 達成したこと・未達のこと

### Step 2: スラッグ・メタデータの決定

以下を決定する:

- `date`: `YYYY-MM-DD` 形式（今日の日付）
- `slug`: セッションを一言で表す英数字ハイフン区切り（例: `add-review-thinking-skill`）
- `title`: セッションタイトル（日本語可）
- `goal`: ゴールの一行サマリー（英語推奨、aggregate-reviews での機械解析用）
- `tags`: セッションで扱った技術・操作のタグリスト（例: `git, file-edit, skill-creation, bash`）
- `outcome`: `success` / `partial` / `failed`
- `project`: プロジェクトルートのディレクトリ名（例: `agent-setting`）。`copy-review.sh` が自動付与するが、スキルでも同値を埋める

出力ディレクトリ名: `{date}-{slug}`

### Step 3: thinking.md 生成

以下のテンプレートで `thinking.md` を作成する:

```markdown
# セッションレビュー: {title}

**日時:** {date}
**目標:** {goal}
**タグ:** {tags}
**結果:** {outcome}

---

## 思考プロセスの流れ

{セッション全体の流れをnarrativeで記述。何をどの順序で考え、どう判断したか。}

## 主要な判断ポイント

| 判断 | 選択肢 | 採用理由 |
|------|--------|----------|
| ... | A vs B | ... |

## 意思決定ログ

| # | 状況 | 選択肢 | 採用 | 情報源 | 自信度 | 事後評価 |
|---|------|--------|------|--------|--------|---------|
| 1 | ... | A / B / C | A | コード読み / ユーザー指示 / 推測 | 高/中/低 | 正解 / 次回はBが良い |

**情報源の種類:**
- `コード読み`: ファイルを実際に読んで確認した
- `ユーザー指示`: ユーザーの明示的な指示に従った
- `推測`: 確認せず推測で判断した
- `慣例/ベストプラクティス`: 一般的な慣習に従った
- `エラーから学習`: 失敗を見て方針を変えた

## 実行したアクション

| # | アクション | ツール | 自動化分類 | 判断理由 |
|---|-----------|--------|-----------|---------|
| 1 | ... | Bash/Edit/Write/Read | 完全自動化可能 / 半自動化可能 / LLM実行 / 要人間判断 / 非再現 | ... |

**自動化分類の定義:**
- **完全自動化可能**: シェルコマンド・ファイル操作・API呼び出し等、入出力が決定的
- **半自動化可能**: パラメータ固定なら実行可能だが、環境依存あり
- **LLM実行**: LLMの生成・分析・判断が主体（プロンプトで再現可能）
- **要人間判断**: ユーザー確認・外部操作・意思決定が必要
- **非再現**: 対話的UI操作・外部サービスの状態変化等

## エラーと対処

{ハマった点・試行錯誤・解決策。なければ「特になし」と記載}

## 判断品質レビュー

- **最も良かった判断:** {何を・なぜ正しく判断できたか}
- **最も改善すべき判断:** {何が足りなかったか、次回どうするか}
- **推測に頼った箇所:** {何があれば確信を持てたか。何を確認すれば推測を排除できたか}

## 学習・知見

{次回に活かせるパターン・ベストプラクティス・注意点}

## 抽出したルール

<!-- aggregate-reviews での横断分析用。確信度が高まったルールをここに蓄積する -->
<!--
  scope : global（~/.claude/配下）/ {project-name}（プロジェクトCLAUDE.md）
  domain: general / git / shell / hook / ... （参照タイミングを示す分野名）
          global + general  → ~/.claude/CLAUDE.md（常時ロード）
          global + git      → ~/.claude/rules/git.md
          global + shell    → ~/.claude/rules/shell.md
          {project} + *     → {project}/CLAUDE.md
-->

| パターン | ルール | 根拠（セッション経験） | 確信度 | scope | domain |
|---------|--------|----------------------|--------|-------|--------|
| {こういう状況では} | {こうする} | {このセッションの経験} | 低（1回） / 中 / 高 | global / {project-name} | general / git / shell / hook / ... |

## review-thinking スキルへの改善提案

このセッションの経験に基づき、review-thinking の SKILL.md を改善するなら:

| 対象ステップ | 現在の記述 | 改善案 | 根拠（このセッションの経験） | 反映状況 |
|------------|-----------|--------|--------------------------|---------|
| ...        | ...       | ...    | ...                      | 未反映 / 反映済み / 却下（理由） |

提案なし: セッションで新たな課題が発見されなかった場合はその旨を記載

## その他設定・スキルへの改善提案

このセッションで気づいた、他のスキルや設定ファイルへの改善提案:

| 対象ファイル | 現在の記述 | 改善案 | 根拠（このセッションの経験） | 反映状況 |
|------------|-----------|--------|--------------------------|---------|
| ...        | ...       | ...    | ...                      | 未反映 / 反映済み / 却下（理由） |

提案なし: 該当なければその旨を記載
```

### Step 4: 自動化可能ステップの洗い出し

reproduce スクリプト生成の前に、Step 3 のアクション表と **意思決定ログ** を元に各アクションを以下の基準で再確認する。
各ステップで「なぜその方法を選んだか・何を却下したか・何が不確かだったか」を意識的に掘り起こすこと。
これが reproduce スクリプトの `[思考]` ブロックの内容になる。

| 分類 | 基準 | スクリプト上の扱い |
|------|------|--------------------|
| **完全自動化可能** | シェルコマンド・ファイル操作・API呼び出し等 | コマンドとして実装 |
| **半自動化可能** | 環境依存あり・パラメータ変数化すれば実行可 | 変数を宣言しコメントで説明 |
| **LLM実行** | LLMの生成・分析・判断が主体 | `# [LLM実行]` + プロンプト記述 |
| **要人間判断** | ユーザー確認・外部操作・意思決定が必要 | `# [要人間判断]` コメントブロック |
| **非再現** | 対話的UI・外部サービスの状態変化等 | 手順説明コメントのみ |

### Step 5: reproduce.{ext} 生成

セッションで実際に実行したアクションの内容・言語・環境に応じて最適な形式を選択する:

**形式の選択基準:**
- Bashコマンドが主体 → `.sh`
- PowerShellコマンドが主体 → `.ps1`
- Pythonコードが主体 → `.py`
- 混在 → `.sh` ベースに他言語のコード片をインライン埋め込み、または複数ファイル

**スクリプトの冒頭に必ず構造化メタデータを埋め込む（aggregate-reviews での機械解析用）:**

```bash
#!/bin/bash
# ============================================================
# [REVIEW-META]
# title: {セッションタイトル}
# date: {YYYY-MM-DD}
# goal: {ゴールの一行サマリー}
# tags: [{タグ1}, {タグ2}, ...]
# outcome: success | partial | failed
# project: {プロジェクトルートのディレクトリ名}
# ============================================================
#
# 使い方: bash reproduce.sh
# ※ [LLM実行] マークのステップはプロンプトを参照してClaude Codeで再現すること
# ※ [要人間判断] マークのステップはコメントを参照して手動で実施すること

set -e

# --- 設定 ---
PROJECT_DIR="${PROJECT_DIR:-$(pwd)}"
```

**各ステップの記述パターン:**

完全自動化可能なステップ:
```bash
# --- Step N: {アクション説明} ---
# [思考] {なぜこのアプローチを選んだか。検討した代替案・却下理由・不確実だった点を含む}
#         例: "A方式とB方式を比較。AはXの懸念があったためBを選択。ただしYは確認せず推測"
# [情報源] コード読み / ユーザー指示 / 推測 / 慣例 / エラーから学習
# [結果] {このステップの期待成果}
{実際のコマンド}
```

LLM実行なステップ:
```bash
# --- Step N: [LLM実行] {アクション説明} ---
# [思考] {なぜLLMに委譲したか。自動化できなかった理由・判断に迷った点・前提とした仮定}
#         例: "ファイル構造から判断できたが、ユーザー意図が不明確だったため一旦確認を優先"
# [情報源] コード読み / ユーザー指示 / 推測 / 慣例 / エラーから学習
# [プロンプト] 以下のコマンドで同等の作業を再現できる:
#   claude "具体的なプロンプト文字列"
#   （またはClaude Codeセッションで手動実行する場合の指示）
# [入力] {参照したファイル・コンテキスト}
# [期待出力] {LLMが生成すべき成果物}
```

要人間判断なステップ:
```bash
# --- Step N: [要人間判断] {アクション説明} ---
# [思考] {なぜ人間の判断が必要だったか。LLMが判断できなかった理由・情報不足の内容}
# [手順] {人間が行うべき操作}
# [判断基準] {何を見て次のアクションを決めるか}
```

**スクリプト記述の方針:**
- ファイル編集（Edit/Write）は内容をインラインで埋め込むか、`cat > file << 'EOF'` 形式で記述
- 環境依存の値（パス・APIキー等）は変数化して冒頭に宣言
- 完全自動化は必須ではない。決定的に再現できる部分はスクリプト化し、そうでない部分は手順として明示

PowerShell（`.ps1`）の場合のメタデータ形式:
```powershell
# ============================================================
# [REVIEW-META]
# title: {セッションタイトル}
# date: {YYYY-MM-DD}
# goal: {ゴールの一行サマリー}
# tags: [{タグ1}, {タグ2}, ...]
# outcome: success | partial | failed
# project: {プロジェクトルートのディレクトリ名}
# ============================================================
#
# 使い方: pwsh reproduce.ps1

Set-StrictMode -Version Latest
$ErrorActionPreference = "Stop"

# --- 設定 ---
$ProjectDir = if ($env:PROJECT_DIR) { $env:PROJECT_DIR } else { Get-Location }
```

### Step 6: ディレクトリ生成・ファイル保存

1. `{project}/reviews/{date}-{slug}/` ディレクトリを作成（Bash `mkdir -p` を使用）
2. `thinking.md` を保存
3. `reproduce.{ext}` を保存
4. スクリプトが `.sh` の場合は `chmod +x reproduce.sh` で実行権限を付与

`dest` へのコピーは `~/.claude/scripts/copy-review.sh` が PostToolUse フックとして自動実行するため、手動コピーは不要。

> **注意:** `copy-review.sh` は `reproduce.*` の書き込みをトリガーにコピーを実行する。
> **thinking.md → reproduce.{ext} の保存順序を変更しないこと。** 順序が逆になると、コピー時に thinking.md が存在しない状態になる。

### Step 7: コピー先の確認（参考情報）

コピーは `~/.claude/scripts/copy-review.sh` が PostToolUse フックとして自動実行する。
スキル側での操作は不要。

設定ファイルの優先順位（フックスクリプトが参照）:
1. プロジェクト内 `.claude/review-thinking.config` の `dest` キー
2. グローバル設定 `~/.claude/review-thinking.config` の `dest` キー

```yaml
# ~/.claude/review-thinking.config（setup.sh が生成）
dest: ~/reviews   # 環境ごとに異なるパスを指定可能
```

### Step 8: 完了レポート

以下のフォーマットでユーザーに報告する:

```
## review-thinking 完了

### 生成ファイル
- reviews/{slug}/thinking.md       — 思考プロセスドキュメント
- reviews/{slug}/reproduce.{ext}   — 再現スクリプト

### メタデータ
- タグ: [{tags}]
- 結果: {outcome}

### 再現実行
```bash
bash reviews/{slug}/reproduce.sh
```
（または pwsh / python に応じたコマンド）

### 蓄積状況
reviews/ ディレクトリ内のレビュー数: {件数}
```

### Step 9: SKILL.md へのフィードバック（任意）

thinking.md の以下2つの改善提案セクションを確認する:

1. **review-thinking スキルへの改善提案** — 提案がある場合、ユーザーに確認する:
   > 「review-thinking SKILL.md への改善提案があります。反映しますか？
   >   対象: {ステップ名}
   >   内容: {改善案の概要}」

2. **その他設定・スキルへの改善提案** — 提案がある場合、ユーザーに確認する:
   > 「{対象ファイル} への改善提案があります。反映しますか？
   >   内容: {改善案の概要}」

各提案について:

**承認された場合:**

1. **agent-setting-path の取得**
   ```bash
   cat ~/.claude/review-thinking.config
   ```
   `agent-setting-path:` の値を読み取る。取得できない場合は:
   > 「`~/.claude/review-thinking.config` に `agent-setting-path:` が設定されていません。
   >   agent-setting リポジトリで `bash setup.sh` を再実行してください。」

2. **ソースファイルを編集**
   対象ファイルを `{agent-setting-path}/claude/skills/{skill-name}/SKILL.md` に解決して Edit する。
   - 解決例: thinking.md の「対象ファイル」列が `review-thinking` や `review-thinking/SKILL.md` → `{agent-setting-path}/claude/skills/review-thinking/SKILL.md`

3. **インストール先に同期**
   ```bash
   cp {agent-setting-path}/claude/skills/{skill}/SKILL.md ~/.claude/skills/{skill}/SKILL.md
   ```
   編集後に thinking.md の反映状況を「反映済み」に更新する。

**拒否された場合** → thinking.md の反映状況を「却下（理由）」に更新して完了とする

**スキップした場合** → 「未反映」のままとする

これにより、review-thinking スキルは実行のたびに自己改善できる。

## 注意事項

- セッションコンテキストが長い場合、全アクションを漏れなく列挙することを優先する
- `thinking.md` は人間が後から読んで判断プロセスを追えることを最優先とする
- `reproduce.{ext}` は完全自動化ではなく「再現性の最大化」を目指す。実行不可能な部分をコメントで明示する方が、省略するより価値がある
- 構造化メタデータ（`[REVIEW-META]`）は必ず埋め込む。将来の `aggregate-reviews` スキルがこれを参照する
